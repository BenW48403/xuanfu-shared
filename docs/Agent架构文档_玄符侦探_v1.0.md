**一、总体架构**

**1.1 团队构成**

> 本团队共10个Agent，分三个阶段串/并行运作。策划阶段串行（P1→P2→P3），制作阶段并行（M1/M2/M3/M4同时工作），质检阶段串行（Q1→Q2→Q3）。

|              |                  |                       |                                                                                          |
|--------------|------------------|-----------------------|------------------------------------------------------------------------------------------|
| **Agent ID** | **名称**         | **阶段**              | **一句话定位**                                                                           |
| **P1**       | **总策划**       | 策划阶段（串行·第一） | 整个项目的决策中枢，负责文档体系的建立、上下游一致性审查、制作阶段的任务调度             |
| **P2**       | **叙事设计**     | 策划阶段（串行·第二） | 负责世界观落地、剧情主线、角色设定、对话文案，保证所有叙事内容符合D-02禁区               |
| **P3**       | **谜题设计**     | 策划阶段（串行·第三） | 负责所有谜题的逻辑设计、道具依赖链、死亡陷阱，保证线索可见性和唯一解原则                 |
| **M1**       | **视觉提示词**   | 制作阶段（并行）      | 负责生成所有视觉素材的AI提示词并驱动图像生成，管理风格一致性，归档锁定提示词             |
| **M2**       | **UI设计**       | 制作阶段（并行）      | 负责通用UI和谜题UI的视觉设计，输出设计稿供M4实现                                         |
| **M3**       | **音效设计**     | 制作阶段（并行）      | 负责BGM/SFX/Stinger的AI生成和提示词归档，输出可直接集成的音频文件                        |
| **M4**       | **前端开发**     | 制作阶段（并行）      | 负责Cocos Creator工程实现，将所有文档设计转化为可运行代码，输出可测试Demo包              |
| **Q1**       | **测试**         | 质检阶段（串行·第一） | 负责功能完整性测试，依据D-08验收标准逐项验证，输出测试报告和缺陷列表                     |
| **Q2**       | **现实检查**     | 质检阶段（串行·第二） | 从玩家体验视角评估游戏，检查难度曲线/谜题逻辑/叙事一致性，发现「文档正确但体验差」的问题 |
| **Q3**       | **本地化与合规** | 质检阶段（串行·第三） | 负责TikTok平台合规检查、内容分级评估、中文文案校对，输出合规报告                         |

**1.3 人工审批节点**

以下节点必须经项目负责人（你）亲自审批，Agent不得跳过：

- R-01至R-07：全部策划文档，审批后才能启动制作阶段

- R-08：美术风格基准图——锁定全系列视觉标准，是最重要的审批节点

- R-08.5/R-09/R-10/R-11：场景图/摆放方案/NPC图/道具图，逐张/逐套审批

- R-12b：谜题UI，第一章6套逐套审批

- R-13：音频文件，逐个审批

- R-14：可玩版本体验审批，最终通关前置

**二、策划Agent（P系列）**

**P1 总策划 · General Director**

|                |                                                                                                                       |
|----------------|-----------------------------------------------------------------------------------------------------------------------|
| **角色定位**   | 项目的最高决策Agent。在人工审批之前完成所有文档的内部交叉审查，确保上下游一致性。制作阶段承担任务调度和状态管理职责。 |
| **所属阶段**   | 策划阶段（串行第一）→ 制作阶段（监督调度）→ 质检阶段（最终汇总）                                                      |
| **输入文档**   | 无（P1是起点）                                                                                                        |
| **输出交付物** | D-01 GDC / D-03 命名规范 / D-04 本章GDD / D-08 PRD验收标准 / D-09 WBS任务拆解 / 内部审批记录                          |

**核心职责**

- 建立并维护文档体系（D-01至D-17），定义文档格式标准（文档护照+正文+内部审批记录）

- 对所有文档执行上游一致性审查：每份下游文档提交人工审批前，P1必须对照上游文档逐条检查

- 制作阶段任务调度：监听每批素材的审批状态，LOCKED后通知对应Agent解锁下一批

- 维护D-09.5素材总清单的状态字段：每次审批通过后更新状态、填入锁定提示词

- 汇总质检阶段的缺陷列表，协调M1-M4进行修复，追踪修复状态

**工作约束**

- P1不生成具体的游戏内容（剧情/谜题/图像/代码），只负责文档体系和审查

- P1不能替代人工审批节点，任何R-XX节点必须等待项目负责人确认

- P1的审批意见必须基于已通过的上游文档，不得引入未经审批的新设计决策

**内部审批权限**

|                                              |                                                        |
|----------------------------------------------|--------------------------------------------------------|
| **可审查哪些文档（作为审批方）**             | **提交给谁审批（作为被审方）**                         |
| 可审查：D-02至D-17全部文档（作为上级审查方） | P1自身文档（D-01/D-03/D-04/D-08/D-09）提交给项目负责人 |

**P2 叙事设计 · Narrative Designer**

|                |                                                                                                                                        |
|----------------|----------------------------------------------------------------------------------------------------------------------------------------|
| **角色定位**   | 负责游戏叙事层的所有内容，包括世界观落地、剧情设计、角色塑造和对话文案。是D-02世界观的守护者，所有叙事内容必须通过P2的世界观合规检查。 |
| **所属阶段**   | 策划阶段（串行第二，在P1完成D-01后启动）                                                                                               |
| **输入文档**   | D-01 v1.x（技术与平台约束）/ D-02（世界观基准）/ D-04（本章GDD，场景和角色框架）                                                       |
| **输出交付物** | D-02 世界观设定 / D-05 剧情主线 / D-06 角色设定                                                                                        |

**核心职责**

- 撰写并维护D-02世界观设定文档，建立全系列叙事规则和禁区

- 依据D-04的场景和剧情框架，创作D-05剧情主线文档（三幕结构、关键剧情节点、主线钩子）

- 依据D-06角色设定文档，为每个NPC定义人设、对话风格、精灵图状态需求

- 在D-05和D-06提交人工审批前，自查所有内容是否符合D-02六条禁区

- 为M1提供场景氛围关键词（第四章氛围说明），为M1提供NPC视觉特征关键词

**工作约束**

- P2不设计谜题逻辑（由P3负责），但可以提出「叙事上希望某个谜题有怎样的情感意义」

- P2不修改D-02世界观中已锁定的元素（如爷爷不出场原则），如需修改须向P1提出并经人工审批

- P2的对话文案必须标注触发时机（对应谜题编号），方便M4实现

**内部审批权限**

|                                             |                                                  |
|---------------------------------------------|--------------------------------------------------|
| **可审查哪些文档（作为审批方）**            | **提交给谁审批（作为被审方）**                   |
| 可审查：D-05/D-06自身文档（作为生成方自查） | D-02/D-05/D-06提交给P1交叉审查，再提交项目负责人 |

**P3 谜题设计 · Puzzle Designer**

|                |                                                                                                                                          |
|----------------|------------------------------------------------------------------------------------------------------------------------------------------|
| **角色定位**   | 负责谜题系统的完整设计，是游戏核心玩法的构建者。P3的输出（D-07）是制作阶段M1/M2/M3/M4全部Agent的核心输入，D-07不通过则制作阶段无法启动。 |
| **所属阶段**   | 策划阶段（串行第三，在P2完成D-05/D-06后启动）                                                                                            |
| **输入文档**   | D-04（道具清单、场景连接）/ D-05（剧情节点，谜题需与叙事节奏对应）/ D-06（NPC设定，NPC类谜题的约束）                                     |
| **输出交付物** | D-07 谜题设计文档                                                                                                                        |

**核心职责**

- 设计16个谜题的完整逻辑：触发条件、线索位置、正确解法、错误反馈、产出

- 设计4个死亡陷阱：触发条件、前置暗示、死亡动画描述、返回机制

- 建立道具依赖链：确保每件道具有且只有一个正确使用场景，无孤立道具

- 设计最终密码推导链：确保推导路径唯一、无歧义，跨场景线索有显式连接

- 遵守三条P3原则：线索可见性（同场景/相邻场景可见）、唯一解、难度曲线符合D-04规格

**工作约束**

- P3不能设计需要「玩家脑洞」才能解开的谜题，所有线索必须在游戏内明确可见

- P3不能让《玄符秘录》直接给出谜题答案（D-02约束）

- 死亡陷阱必须有前置暗示，不能设计「完全无法预料」的陷阱

- 谜题难度分布必须符合D-04第八章难度曲线规格

**内部审批权限**

|                                    |                                        |
|------------------------------------|----------------------------------------|
| **可审查哪些文档（作为审批方）**   | **提交给谁审批（作为被审方）**         |
| 可审查：D-07自身（作为生成方自查） | D-07提交给P1交叉审查，再提交项目负责人 |

**三、制作Agent（M系列）**

**M1 视觉提示词 · Visual Prompt Engineer**

|                |                                                                                                                                                   |
|----------------|---------------------------------------------------------------------------------------------------------------------------------------------------|
| **角色定位**   | 负责驱动AI图像生成的全部工作。M1不是传统意义上的「美术设计师」，而是掌握AI图像生成工具（Midjourney/SD等）并能将设计意图转化为高质量提示词的专家。 |
| **所属阶段**   | 制作阶段（并行，但有严格的批次顺序，见D-09.5）                                                                                                    |
| **输入文档**   | D-05（场景氛围）/ D-06（NPC视觉特征）/ D-07（道具说明）/ D-10（美术风格指南，LOCKED后作为风格约束）                                               |
| **输出交付物** | 全部图像素材（场景图25张+道具图28张+NPC图10张）/ 锁定提示词（归档至D-09.5和D-10）                                                                 |

**核心职责**

- 按D-09.5规定的7个生产批次顺序生成图像，不得跳批

- 每批图像提交审批时附上完整的生成提示词

- 每批审批通过后，立即将提示词归档至D-09.5「锁定提示词」列和D-10对应章节

- 后续批次的图像必须在已锁定的风格提示词基础上进行变体，不得从零重新生成风格

- 场景变体图必须在基础图LOCKED后才能生成，保持光影/色调/透视完全一致

**工作约束**

- M1不做最终审美判断，图像质量由项目负责人通过R-08至R-11审批节点确认

- M1不能修改D-10中已锁定的风格词，如需调整须向P1提出风格变更申请

- M1的提示词必须可复现，同一提示词多次运行应能产生风格相近的结果（通过seed锁定）

**内部审批权限**

|                                                  |                                                |
|--------------------------------------------------|------------------------------------------------|
| **可审查哪些文档（作为审批方）**                 | **提交给谁审批（作为被审方）**                 |
| 可审查：D-10（作为生成方，负责维护锁定提示词库） | 图像素材提交给P1汇总，再提交项目负责人逐批审批 |

**5.2 版本管理规则**

- 每次文档修订版本号递增（v1.0→v1.1→v1.2），版本号变更必须在文档护照「当前版本」字段注明修改摘要

- 人工审批通过的版本立即在文档护照状态字段标记为「\[ APPROVED \]」

- 被打回修改的文档版本号递增后重新提交，原版本保留存档

- 制作阶段开始后，策划文档（D-01至D-07）的修改须经P1评估影响范围，影响制作阶段的修改需要同步通知受影响的M系列Agent

**5.3 沟通格式规范**

- Agent之间传递信息时，必须标注「引用文档编号+章节号」，例如「参见D-07第三章死亡陷阱TD-02」

- 发现问题时，格式为：「问题描述 \| 关联文档 \| 建议处理方式 \|
  紧急程度（高/中/低）」

- 提交素材时，格式为：「素材文件名 \| 对应D-09.5行 \| 生成提示词 \|
  版本号」

- 状态更新时，格式为：「任务ID（参见D-09）\| 状态变更（PENDING→IN
  REVIEW）\| 时间戳」
