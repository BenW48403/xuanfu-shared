**一、技术架构**

**1.1 整体架构（v1.1更新）**

v1.1更新：平台层从“TikTok Minis”扩展为多平台支持；存档层从直接调用tt
API改为通过PlatformAdapter统一接口；新增跨平台抽象层。

|          |                                                                                       |
|----------|---------------------------------------------------------------------------------------|
| **层级** | **说明**                                                                              |
| 引擎     | Cocos Creator 3.x，2D模式，TypeScript                                                 |
| 平台     | 主平台：TikTok Minis（抖音小游戏）；扩展平台：微信小游戏、H5（通过跨平台抽象层支持）  |
| 场景管理 | 单Scene多Node架构：主Scene常驻，各游戏场景作为Prefab按需加载/卸载                     |
| 状态管理 | GameStateManager单例：管理全局游戏状态（当前场景/道具栏/已完成谜题/NPC状态/提示状态） |
| 数据层   | JSON配置文件驱动：道具/谜题/NPC/提示热点全部外置配置，代码与数据分离                  |
| 存档     | 跨平台存档适配层：通过PlatformAdapter统一接口，底层按平台切换tt/wx/localStorage实现   |
| 跨平台层 | PlatformAdapter抽象层：统一封装存档/登录/广告三类平台API，游戏逻辑不直接调用平台SDK   |

**1.2 核心数据结构**

GameState（全局状态对象）：

|                 |                                 |                                          |
|-----------------|---------------------------------|------------------------------------------|
| **字段**        | **类型**                        | **说明**                                 |
| currentScene    | string                          | 当前所在场景ID                           |
| inventory       | ItemState\[\]（最多8个）        | 道具栏，包含道具ID和是否为主线道具       |
| puzzleStates    | Map\<PuzzleID, PuzzleState\>    | 16个谜题的状态：IDLE/ACTIVE/COMPLETE     |
| npcStates       | Map\<NPCID, NPCState\>          | 3个NPC的状态：IDLE/TALKING/COMPLETE/GONE |
| sceneStates     | Map\<SceneID, SceneVariant\>    | 场景变体状态（如门开/关，展柜开/关）     |
| hintStates      | Map\<HintHotspotID, HintState\> | 每个热点的提示解锁状态（各层是否已解锁） |
| playTime        | number（秒）                    | 累计游玩时长                             |
| chapterComplete | boolean                         | 章节是否通关                             |

**1.3 提示系统数据结构**

提示系统采用配置驱动设计，热点数量和提示内容测试阶段由项目负责人填入JSON配置，无需修改代码。

HintConfig（提示配置，每个场景一个JSON文件）：

|                |                        |                                                        |
|----------------|------------------------|--------------------------------------------------------|
| **字段**       | **类型**               | **说明**                                               |
| hotspotId      | string                 | 热点唯一ID，格式：hint\_{sceneId}\_{index}             |
| linkedPuzzleId | string                 | 关联的谜题或道具ID，当该谜题/道具完成后热点消失        |
| position       | {x: number, y: number} | 热点在场景中的显示坐标（1920×1080坐标系）              |
| hints          | HintItem\[\]           | 提示项目数组，按层级排列（测试阶段填入，初始为空数组） |

HintItem（单条提示项）：

|              |                    |                                            |
|--------------|--------------------|--------------------------------------------|
| **字段**     | **类型**           | **说明**                                   |
| type         | 'image' \| 'video' | 提示内容类型                               |
| resourcePath | string             | 资源路径，空字符串时显示占位图             |
| requireAd    | boolean            | 是否需要看广告才能解锁（第一层通常为true） |
| unlocked     | boolean            | 当前玩家是否已解锁（存入hintStates）       |

**二、谜题状态机设计**

**2.1 谜题状态流转**

\- IDLE：谜题未触发，热点存在但无法直接与谜题交互

\- ACTIVE：玩家触发谜题（点击热点/使用道具），进入谜题界面或对话

\-
COMPLETE：谜题完成，触发产出（解锁场景/生成道具/推进NPC状态），热点消失

\- 状态流转：IDLE → ACTIVE（触发条件满足）→ COMPLETE（正确解答）

\-
错误操作：ACTIVE状态下错误操作触发错误反馈，不改变状态（除死亡陷阱外）

**2.2 道具依赖检查**

\- 使用道具前系统检查：道具是否在道具栏 + 目标热点是否接受该道具类型

\- 道具组合检查：选中道具A后点击道具B，系统查配置表是否存在A+B的组合规则

\- 主线道具保护：T-12铜牌在道具栏中标记isMainline=true，丢弃操作直接拦截

**三、场景管理**

**3.1 场景切换**

\- 场景切换使用淡入淡出动画（0.3秒黑屏过渡）

\- 切换时自动保存GameState（通过PlatformAdapter.storage.set调用）

\- 每个场景Prefab独立管理其热点列表和状态变体

\-
场景内可交互热点在SceneComponent中注册，进入提示模式时统一查询未完成热点

**3.2 场景状态变体**

\- 每个场景有一个SceneVariant枚举，定义该场景的所有可能状态

\-
变体切换：修改sceneStates中对应的variant值，场景重新加载时读取正确背景图

\- 示例：保安室 → SecurityRoomVariant.DOOR_CLOSED / DOOR_OPEN

**四、性能与包体控制**

\- 包体目标≤50MB，图片使用WebP格式压缩（质量85%），音频使用流式加载

\- 场景背景图不预加载全部，按需加载当前+相邻场景

\- 道具图标128px版本全部预加载，512px详情版按需加载

\- 内存占用目标：运行时峰值≤150MB（适配中低端Android机型）

**五、跨平台抽象层设计（v1.1新增）**

v1.1新增本章节。游戏逻辑不得直接调用任何平台SDK（tt.\*/wx.\*/localStorage），必须经由PlatformAdapter统一接口调用。切换平台时只需替换Adapter实现，不需修改游戏逻辑代码。

**5.1 设计原则**

\-
单一接口原则：所有平台相关API统一收口到PlatformAdapter，游戏层只调用Adapter接口

\- 零耦合原则：游戏逻辑代码中不得出现tt./wx./localStorage等平台特有调用

\- 运行时注入原则：游戏启动时根据构建目标自动注入对应平台的Adapter实现

\- 最小差异原则：只封装实际存在差异的API（存档/登录/广告），不过度抽象

**5.2 PlatformAdapter接口定义**

|                         |                            |                         |                                                                                           |
|-------------------------|----------------------------|-------------------------|-------------------------------------------------------------------------------------------|
| **接口方法**            | **参数**                   | **返回值**              | **说明**                                                                                  |
| storage.set(key, value) | key: string, value: string | Promise\<void\>         | 存储数据。抖音调用tt.setStorage，微信调用wx.setStorage，H5调用localStorage.setItem        |
| storage.get(key)        | key: string                | Promise\<string\|null\> | 读取数据。各平台对应get方法                                                               |
| storage.remove(key)     | key: string                | Promise\<void\>         | 删除数据                                                                                  |
| auth.login()            | —                          | Promise\<UserInfo\>     | 静默登录。抖音调用tt.login，微信调用wx.login，H5使用游客模式                              |
| auth.getUserId()        | —                          | string                  | 获取当前用户唯一ID                                                                        |
| ad.showRewardedAd()     | —                          | Promise\<boolean\>      | 展示激励视频广告。抖音用穿山甲SDK，微信用wx.createRewardedVideoAd，H5返回true（跳过广告） |
| ad.isMonthCardActive()  | —                          | Promise\<boolean\>      | 检查月卡状态。各平台对应支付查询接口                                                      |
| platform.getType()      | —                          | PlatformType            | 返回当前平台标识：'tiktok' \| 'wechat' \| 'h5'                                            |
| platform.vibrate()      | —                          | void                    | 触发短震动（错误反馈）。各平台对应振动API                                                 |

**5.3 各平台实现对照表**

|              |                |                               |                    |                          |                        |
|--------------|----------------|-------------------------------|--------------------|--------------------------|------------------------|
| **平台**     | **构建目标**   | **存档实现**                  | **登录实现**       | **广告实现**             | **备注**               |
| 抖音小游戏   | 字节跳动小游戏 | tt.setStorage / tt.getStorage | tt.login 静默登录  | 穿山甲激励视频SDK        | 主平台，D-01首发目标   |
| TikTok Minis | Web Mobile     | TikTok Storage API            | TikTok Minis SDK   | TikTok广告SDK            | 海外版，API与抖音类似  |
| 微信小游戏   | 微信小游戏     | wx.setStorage / wx.getStorage | wx.login 静默登录  | wx.createRewardedVideoAd | 国内扩展平台           |
| H5网页版     | Web Mobile     | localStorage                  | 游客模式（无登录） | 跳过广告（返回true）     | 最早验证渠道，无需审核 |

**5.4 构建与发行流程**

\- 抹音小游戏：Cocos Creator选择“字节跳动小游戏”构建目标 → 配置AppID →
构建 → 上传字节跳动开放平台 → 提交审核

\- 微信小游戏：Cocos Creator选择“微信小游戏”构建目标 → 配置微信AppID →
构建 → 微信开发者工具上传 → 提交审核

\- TikTok Minis：Cocos Creator选择Web Mobile构建目标 → 接入TikTok Minis
SDK → 打包上传 → 提交审核

\- H5网页版：Cocos Creator选择Web Mobile构建目标 → 部署到Web服务器 →
无需审核，可作为最早验证渠道

**5.5 开发约束**

\-
M4开发时必须使用PlatformAdapter接口，代码审查时检查是否存在直接的平台API调用

\-
第一章开发时仅需实现TikTokAdapter（主平台）和H5Adapter（验证用），WeChatAdapter在扩展期实现

\-
新增平台时只需新建一个Adapter实现类，实现PlatformAdapter接口的全部方法即可
