## Phase 0 — 环境稳定化（预计 0.5–1 天）

**逻辑**：OpenClaw 原生 Windows 有大量已知 bug（Gateway 无响应、spawn EINVAL、Scheduled Task 崩溃、CVE-2026-22176 安全漏洞）。社区和官方文档一致推荐 WSL2。如果不先解决环境问题，后续所有配置都建在不稳定的地基上。

### 0.1 安装 WSL2 + Ubuntu 24.04
PowerShell 运行 `wsl --install`，重启后创建 Linux 用户。启用 systemd（编辑 `/etc/wsl.conf`）。
> ⚠️ 风险：部分机器 BIOS 未开 Hyper-V，需进 BIOS 设置。

### 0.2 在 WSL2 内重新安装 OpenClaw
WSL2 Ubuntu 终端内运行 npm 安装 + onboard 向导。将原生 Windows 的配置迁移过来（复制 `openclaw.json`，重新填入 GLM API key）。验证 `openclaw doctor` 全部通过。
*依赖 → 0.1 完成*

### 0.3 冒烟测试：单 Agent 对话验证
在 CLI 内与默认 Agent 对话，确认 GLM-5-turbo 响应正常、Gateway 稳定运行、无 fetch-timeout 错误。
*依赖 → 0.2 完成*
> ✅ 风险：低，WSL2 内 OpenClaw 稳定性有大量社区验证。

## Phase 1 — 通讯通道搭建（预计 1–2 天）

**逻辑**：你的核心需求是"通过聊天窗口与 Agent 沟通"。通道不通，后面所有 Agent 都无法使用。飞书是你选择的主通道，且 OpenClaw ≥ 2026.2 已内置飞书插件，你的 5.12 版本完全支持。

### 1.1 飞书开放平台创建企业应用
登录 open.feishu.cn → 创建企业应用 → 启用机器人能力 → 复制 App ID 和 App Secret → 导入权限 JSON → 选择 WebSocket 长连接模式 → 发布应用。个人飞书账号即可，无需企业认证。
> ⚠️ 风险：飞书 API 月度免费额度 50,000 次，OpenClaw 每 60 秒 health check 消耗约 27,000 次/月。单机单应用不会超限，但需注意。

### 1.2 OpenClaw 配置飞书通道
WSL2 内运行 `openclaw channels add` → 选 Feishu → 粘贴凭证 → `openclaw gateway restart`。然后在飞书内给机器人发消息，检查是否收到回复。
*依赖 → 0.3 + 1.1 完成*

### 1.3 冒烟测试：飞书 ↔ Agent 对话
在飞书私聊中向机器人发送消息，确认：消息能收到 → Agent 能回复 → 中文显示正常 → 流式输出正常（interactive card）。
*依赖 → 1.2 完成*

## Phase 2 — 多 Agent 基础架构（预计 2–3 天）

**逻辑**：通道通了之后，开始搭建多 Agent 团队。不需要一次搭 10 个——先搭 P1 总策划 + M4 代码 + M1 视觉三个核心 Agent，覆盖策划→美术→代码的最小闭环。其余 Agent 在流程跑通后按需补充。

### 2.1 创建飞书群组结构
在飞书创建群组：①「P1-总策划」群 ②「M1-视觉」群 ③「M4-开发」群。将机器人加入所有群。记录每个群的 `oc_` 开头的 chat_id。

### 2.2 配置三个独立 Agent
使用 `openclaw agents add` 命令创建 p1-planner、m1-visual、m4-coder 三个 Agent。每个 Agent 配置独立的 workspace 目录、独立的系统提示词（persona）。将你的 D-01~D-17 文档核心内容写入每个 Agent 的 workspace 作为上下文。
*依赖 → 1.3 + 2.1 完成*

### 2.3 配置 binding 路由规则
在 `openclaw.json` 的 bindings 中，将 P1 群 → p1-planner Agent，M1 群 → m1-visual Agent，M4 群 → m4-coder Agent。验证在不同群发消息时，正确的 Agent 回复。
*依赖 → 2.2 完成*

### 2.4 配置 Agent 间通信
启用 agentToAgent 工具，设置 allow 列表。配置共享文件目录（如 `/home/user/game-project/shared/`）作为素材传递通道。测试 P1 能否通过 `sessions_send` 给 M1 发指令。
*依赖 → 2.3 完成*
> ⚠️ 风险：Agent 间通信是社区方案，非官方原生功能，可能需要调试。

### 2.5 冒烟测试：多 Agent 协作
在 P1 群下达指令："请 M1 生成一张测试图片的提示词"。验证 P1 是否能把任务转发给 M1，M1 是否能在 M1 群回复结果。
*依赖 → 2.4 完成*

## Phase 3 — 工具链接入（预计 3–5 天）

**逻辑**：Agent 能对话了，但还不能"干活"。这一阶段给 M1 接入图像生成能力、给 M4 接入编码能力。这是验证阶段最核心的环节——证明 Agent 不只是聊天，而是能产出实际素材和代码。

### 3.1 M1 视觉 Agent 接入图像生成
方案 A（推荐起步）：你已有 ChatGPT Plus，让 M1 生成提示词后你手动在 ChatGPT 生成图片。方案 B（自动化）：通过 OpenClaw 的 MCP 接入 GPT Image API 或 Flux API，M1 自动调用生成。先用方案 A 验证提示词质量，再升级到方案 B。
*依赖 → 2.5 完成*

### 3.2 M4 代码 Agent 接入编码工具
通过 OpenClaw 内置的 claude-cli 后端或 Claw Orchestrator 接入 Claude Code / OpenCode CLI。在 WSL2 内安装 OpenCode，配置 GLM-5-turbo 作为编码模型。创建 Cocos Creator 项目目录，让 M4 Agent 能在其中读写代码。
*依赖 → 2.5 完成*
> ⚠️ 风险：这是最复杂的集成点，GLM-5-turbo 的编码能力可能不足，需要评估是否切换为更强模型。

### 3.3 M3 音效 Agent 接入音频工具
SFX 部分：通过 MCP 接入 Freesound API（免费）搜索音效。BGM/Stinger 部分：M3 生成 Suno 提示词，你手动在 Suno 免费账户生成，或接入 Suno 第三方 API。
*依赖 → 2.5 完成*

### 3.4 端到端冒烟测试
选一个最小的子任务做端到端验证。例如：在 P1 群说"请 M1 生成入口大厅场景图的提示词" → M1 生成提示词 → 你把提示词生成图片 → 图片存入共享目录 → 在 M4 群说"请确认场景图已到位" → M4 确认能读取。
*依赖 → 3.1 + 3.2 + 3.3 完成*

## Phase 4 — 故事与文档打磨（预计 5–7 天）

**逻辑**：工具链跑通后，开始用 Agent 团队做实际工作。第一件事不是直接生产素材，而是先把故事线和文档打磨到位——因为所有素材和代码都依赖于策划文档。去掉"玄符秘录"约束、完善 9 章主线、打磨世界观，然后级联更新 D-02~D-09.5 全套文档。

### 4.1 与 P1/P2 重新设计世界观和 9 章主线
在 P1 群与总策划 Agent 讨论：去掉玄符秘录约束、确定 9 章的主题和场景设定、重新定义主线悬念结构。P2 叙事 Agent 负责世界观一致性审查。

### 4.2 级联更新全套文档
D-02 世界观 → D-04 本章 GDD → D-05 剧情 → D-06 角色 → D-07 谜题 → D-08 PRD → D-09.5 素材清单。每份文档更新后由 P1 交叉审查一致性。
*依赖 → 4.1 完成*

### 4.3 你的人工审批
根据人工验收机制 v2.0 的流程，逐份审批更新后的文档。重点审查：故事是否有吸引力、谜题是否有趣、9 章结构是否合理。
*依赖 → 4.2 完成*

## Phase 5 — 第一章素材生产与开发（预计 2–4 周）

**逻辑**：文档定稿后，进入实际生产。按 D-09 WBS 的任务拆解，M1/M2/M3 并行生产素材，M4 同步开发核心框架代码。你作为项目负责人在飞书群里逐项审批素材。这是耗时最长的阶段。

### 5.1 M4 搭建项目骨架
M4 在 Cocos Creator 项目中实现：GameStateManager、PlatformAdapter、SceneManager、HotspotComponent 等核心框架。使用 D-13 v1.1 的架构设计和 D-15 的功能需求清单。

### 5.2 M1 批量生成场景图和道具图
按 D-09.5 素材清单和 D-10 美术风格指南，逐批生成。每批提交你审批，通过后锁定提示词归档。先基础图再变体图（D-10 v1.1 第七章规范）。

### 5.3 M2 生成 UI 元素 + M3 生成音频
M2 按 D-11 UI 视觉规范生成道具栏、对话框等通用 UI。M3 按 D-12 音频设计文档生成 BGM/SFX/Stinger。与 M1 并行进行。

### 5.4 M4 集成素材 + 实现谜题逻辑
M1/M2/M3 的审批通过素材存入共享目录后，M4 集成到 Cocos Creator 项目中。按 D-14 数据表规范填充 JSON 配置文件，按 D-07 实现 16 个谜题的交互逻辑。
*依赖 → 5.1 + 5.2 + 5.3 的素材陆续到位*

### 5.5 Q1 测试 + 你的最终验收
按 D-16 测试用例执行测试。先出 H5 网页版（无需审核，最快验证），你在浏览器里试玩完整流程。修复问题后打包抖音小游戏版本。
*依赖 → 5.4 完成*

## Phase 6 — 复制到第 2–9 章 + 引入 Hermes（后续）

**逻辑**：第一章跑通后，你已经有了一条经过验证的流水线。此时引入 Hermes Agent 作为执行层，让它从第一章的经验中自动学习。后续章节的生产效率会随着 Hermes 的学习循环逐章提升。

### 6.1 第一章复盘：记录流水线优化点
哪些步骤最耗时？哪些 Agent 产出质量不稳定？哪些环节需要频繁人工介入？这份记录就是第二步 SaaS 产品化的需求文档。

### 6.2 引入 Hermes，搭建 OpenClaw + Hermes 双框架
将执行类 Agent（M1/M2/M3/M4）迁移到 Hermes，利用其学习循环自动优化。OpenClaw 保留编排调度角色（P1 + 路由 + 通道管理）。

### 6.3 批量生产第 2–9 章
每章重复 Phase 4 → Phase 5 的流程，但效率逐章提升。同时收集用户反馈，为 SaaS 产品化积累需求。
