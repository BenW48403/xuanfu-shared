# Demo-1（拜堂克隆）· 导航与缘起

> 给任何首次读到本目录的人或 AI:这份 README 把"从原项目到 Demo-1"的脉络串起来,读完就懂这里每份文件为什么存在、怎么用。

## 1. 原项目
《玄符侦探》:一人工作室开发的中式恐怖点击解谜手游(Cocos Creator 3.8.8 + TypeScript,目标抖音/微信小游戏 + H5)。9 章、镇阴博物馆题材。
- **原 D 系文档(docs/ 下 D-01~D-17)= 原创游戏的设计**,是规划/设计 Agent(P2 剧情、P3 谜题)的战场。**本 Demo 一字不动它们。**
- 长期愿景:把"多 Agent 制片流水线"产品化成 SaaS——任何人给一个故事方向,产出一整个解谜小游戏。
- Agent 架构(docs/Agent架构):P1 统合、M1 图、M3 音效、M4 代码(= 带 Superpowers+ECC 的 Claude Code 编码 agent)等。

## 2. 为什么先做 Demo-1
要验证的核心命题是:**"拆解 → M1/M3/M4 生产 → M4 装配 → 可玩章节"这条多 Agent 流水线,能不能端到端贯通**,并测单章吞吐(回答"九章做不做得完")。
- 为隔离变量,Demo-1 选**忠实复刻**竞品《纸嫁衣1》第一章「拜堂」:机制/谜题结构照搬,美术/音效自做(相似即可),对白只用占位词。
- **克隆 = 纯誊写,没有"设计"环节**,所以 P2/P3(设计 Agent)不进本回路(选项A)。Demo-1 只验**生产 + 装配**半边。设计能力留到"换原创内容"阶段再验。

## 3. Demo-D 系 vs 原 D 系
本目录的文档是**拜堂克隆的"平行 Demo-D 系"**:与原 D 系同格式,供现有 Agent 直接消费,但描述的是克隆标的而非原创游戏。**两套并存、互不覆盖。**

## 4. 本目录文件导航
- **Demo-D系_拜堂_v3.2正式版.md** = 唯一权威建造蓝图。含:场景导航 / 交互精灵(背景base+叠加精灵模型)/ 道具状态机 / 4 谜题+解锁前置门控 / 过场(占位词)/ D-09.5 完整素材表 / 音效 / 数据槽 / 闭合校验。
  - 喂法:D-04/D-07 → M4 装配;D-09.5 → M1 出图;D-12 → M3 音效;过场词 = 占位,P2 换原创时替换。
- **截图_asset映射_拜堂.csv** = 108 张顺序截图 → asset_id 映射(标"重复可删"的为冗余帧)。是参考包改名的依据。
- **M1/ref_named/** = M1 参考帧包(≈99 张 `ref_<asset_id>.jpg`,由上表改名而来)。**仅供 M1 模仿构图/光照/氛围,不照搬原画。**

## 5. 关键设计模型(读文档前先知道)
- **背景 base + inpaint 叠加精灵**:一张干净背景 + 独立多态精灵;M1 用图生图原位重绘出精灵态(光照一致、加法非乘法);精灵摆位在 Cocos(M4)阶段手配。
- **解锁前置门控**:每个谜题有"前置线索集",未集齐则答案正确也不响应(杜绝盲猜)。
- **线索卡 / 查看层**:线索卡(如地支数字书)M1 只出原貌;"查看层"(黑底+白情节文字+蓝线索文字)由 **M4 程序直出**,非 M1 素材。

## 6. IP 边界（务必遵守）
机制/谜题结构/依赖逻辑照搬(玩法不受版权保护);美术、音效自做、风格相似即可,**不照搬原画原音**;**对白只写占位功能词,绝不誊抄原句**(换原创时填镇阴博物馆自己的内容)。
